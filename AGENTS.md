# MetaLoop 项目宪章

本文件约束本仓库中的所有设计、实现、审查和路线决策。若局部方案、历史文档或实现惯性与本文件冲突，以本文件为准，并显式修正冲突。

## 核心思想

> MetaLoop 是一个极小、正交、事件触发的外环控制系统。它通过持久目标、当前状态、反馈纠正和按需独立观察，使强大但存在随机性、局部视野与自我误判的 Agent，在不确定环境中持续收敛到用户锁定的目标；用户只负责指挥意图、真实例外和保留的最终决策。

这句话是 MetaLoop 的最高层产品与架构约束：

- “极小、正交”约束系统规模；
- “事件触发”控制协议成本；
- “外环控制”定义反馈机制；
- “持久目标、当前状态”解决遗忘、恢复和任务切换；
- “按需独立观察”降低同一上下文和同一思路产生的共享盲点；
- “用户锁定目标”定义收敛方向；
- “用户只负责……”定义 Agent 自主性和升级边界。

## 默认信任模型

MetaLoop 默认信任 Agent 是合作的，并且真诚地希望完成用户目标。系统需要处理的不是恶意 Agent，而是可信 Agent 在长时间、不确定环境中不可避免的随机性、上下文压缩、局部视野、遗漏、自我误判和共享盲点。

内核约束用于外部化 Agent 原本需要持续记住的事实，保持反馈闭环和状态连续性；它们不是为了增加审批、限制 Agent 自主性或构建针对 Agent 的零信任安全边界。只有用户或具体领域明确提出对抗性安全需求时，才引入相应威胁模型。

Context identity、host attestation、不可伪造的执行证明和类似认证机制不属于 MetaLoop 默认协议。若宿主愿意提供，它们只能作为可选观察元数据，不能成为普通 Review 或 Tier 3 acceptance 的前置条件。MetaLoop 默认相信 Codex，并通过明确目标、持久状态、结构化反馈和按需复核帮助 Codex 发现遗漏。

## 功能准入规则

后续所有功能必须能够从核心思想推出。一个功能只有在能够实质完成下列至少一项，并且没有引入不成比例的协议成本时，才应进入 MetaLoop：

1. 减少目标、约束、决定、进展或风险的遗忘与遗漏；
2. 消除当前状态、工作归属、证据、反馈或下一步的歧义；
3. 提供能够改变后续行动的有效反馈、纠正或按需独立观察；
4. 降低用户在协议操作、重复说明、状态恢复和常规授权上的负担。

设计或审查新功能时，必须回答：

- 它解决了哪一种具体失败模式？
- 为什么现有正交机制不能解决？
- 什么事件触发它，什么稳定条件使它保持休眠或退出？
- 它如何帮助 Agent 收敛，而不是把流程本身变成工作目标？
- 它是否增加新的状态权威、任务本体、后台运行时或用户操作？如果是，收益是否足以证明这种复杂度？

不能减少遗漏、消除状态歧义、提供有效反馈或降低用户负担的机制，不应进入 MetaLoop。仅仅让记录更完整、流程更正式、控制看起来更严格或实现更通用，不构成增加功能的充分理由。

## 实施边界

- 内环稳定时，MetaLoop 外环应保持休眠；只在观察到恢复、归属、验证、语义判断或 authority 风险时激活最小充分控制。
- 实现优先、主张后置。assurance 约束最终 completion claim，不因任务涉及架构、跨模块或 MetaLoop 自身，就让整个实现阶段进入高保证流程。
- Tier 1 正常路径只暴露 `task begin` 与可重复执行的 `attempt finish`；常见 workspace 继承、reconciliation、Evidence 绑定、seal 和 mechanical verify 由协议自动吸收。
- Git 负责 workspace change truth，SQLite 负责 protocol state truth，项目文档负责 architecture content truth，验证与按需 Review 负责 completion truth；不要复制权威。
- 不建立第二套 Task ontology、项目管理器、Agent scheduler、常驻 daemon、transcript store 或 vector memory。
- 独立 Review 的目的，是通过新上下文降低相关性盲点；不是表达对 Worker 的不信任。
- 普通修复、测试失败、review follow-up 和同一 acceptance target 下的重试留在同一 Task。
- 用户只处理目标与边界、真实例外、外部资源或权限，以及用户明确保留的最终决策；常规推进和内部协议由 Agent 自主完成。
- 普通 durable Task 以两个 Agent-facing lifecycle command 为预算；超过预算时优先简化或恢复已有收口入口，而不是增加记录、命令或用户操作。
- 评价升级是否成功，优先观察遗漏、重复工作、恢复时间、有效反馈和用户中断是否减少，而不是记录数、命令数或流程覆盖率是否增加。

## 决策优先级

发生架构分歧时，依次判断：

1. 是否更可靠地收敛到用户锁定的目标；
2. 是否减少遗漏、状态歧义和共享盲点；
3. 是否降低用户负担并保留 Agent 自主性；
4. 是否保持极小、正交和事件触发；
5. 在以上条件相同的情况下，再比较功能丰富度、通用性和实现便利性。

任何无法通过上述判断的机制，应删除、简化或留在项目外部，而不是因为已经实现就保留。
