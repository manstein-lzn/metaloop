# MetaLoop v3.6 Prompt Compiler Plan

Last updated: 2026-05-04

## Decision

MetaLoop v3.6 introduces prompt compilation in phases. Phase 0 and Phase 1 are deliberately conservative:

- Phase 0: add semantic baseline tests for the current hardcoded runtime prompt builders.
- Phase 1: add a read-only Prompt Pack loader/compiler that can parse and render prompt files, but is not wired into the runtime main path.
- Phase 2: migrate only Co-Design brainstorm runtime to the Prompt Pack loader.
- Phase 3: migrate only Co-Design discovery/interviewer runtime to the Prompt Pack loader.
- Phase 4: migrate only run/redesign focused route prompt to the Prompt Pack loader.

The runtime source of truth remains the Python builders except for `_build_codex_brainstorm_prompt` / `CodexCoDesignBrainstormer`, which compile `prompts/co_design/brainstorm.md`, `_build_codex_interviewer_prompt` / `CodexCoDesignInterviewer`, which compile `prompts/co_design/discovery.md`, and `build_focused_route_prompt`, which compiles `prompts/run/redesign.md`. The main goal prompt, repair, and soft review prompts are not migrated yet.

## Why Not `read_text().replace`

MetaLoop prompts are not simple prose templates. The final work order is a compiled artifact made from three inputs:

- normative prompt instructions from versioned prompt files
- structured MetaLoop state such as MissionSpec, MissionCapsule, GoalContract, VerificationResult, route, role, and attempt state
- schemas and runtime guardrails such as output schemas, locked contract rules, repair attempt discipline, and route enums

A raw `read_text().replace` path would hide too much policy in call sites, make missing state easy to ignore, and make partial migrations hard to audit. The compiler therefore must fail fast on missing metadata, missing variables, empty variables, and unresolved placeholders. It should stay boring and strict before it becomes powerful.

## Three Layers

Prompt Pack:

- Files under `prompts/`.
- Versioned, reviewable, and audit-friendly.
- Contains metadata: `id`, `stage`, `version`, `purpose`, `input_schema`, `output_schema`, `failure_policy`, and `required_variables`.
- Does not execute runtime logic.

Prompt Compiler:

- Loads Prompt Pack files.
- Parses simple front matter without PyYAML.
- Performs strict `{{var_name}}` replacement only.
- Rejects expressions, includes, loops, conditionals, missing variables, empty variables, and unresolved placeholders.
- Returns rendered text plus metadata and sha256.

Runtime Prompt:

- The final work order sent to Codex or another LLM.
- Eventually compiled from Prompt Pack instructions plus structured state and schema payloads.
- After Phase 4, Co-Design brainstorm, Co-Design discovery/interviewer, and run/redesign focused route guidance are compiled from the prompt pack; all other runtime prompts remain generated by Python builders.

## Migration Plan

Phase 0: Semantic Baseline

- Add tests that assert key semantics in current hardcoded prompt builders.
- Avoid full snapshots; check durable invariants such as no execution during design, route enum, locked contract bans, repair attempt index, root cause/hypothesis discipline, and required structured state injection.

Phase 1: Read-Only Loader

- Add `src/metaloop/prompt_pack.py`.
- Parse front matter and render prompt files strictly.
- Add tests for metadata parsing, render success, failure modes, stable hash, and existing prompt file validity.
- Do not wire the loader into the runtime.

Phase 2: Brainstorm Migration

- Status: implemented.
- Co-Design brainstorm migrated first because it is design-time, low blast-radius, and already JSON-shaped.
- `_build_codex_brainstorm_prompt` now renders `prompts/co_design/brainstorm.md` through `metaloop.prompt_pack.render_prompt`.
- Python injects `mission_spec`, `co_design_draft`, and `mission_spec_review` as fenced JSON.
- Prompt rendering failures raise `CoDesignAgentError`; there is no hardcoded fallback.
- Codex unavailable, invalid JSON, and no usable brainstorm options remain fail-fast.

Phase 3: Discovery Migration

- Status: implemented.
- Co-Design discovery/interviewer now renders `prompts/co_design/discovery.md` through `metaloop.prompt_pack.render_prompt`.
- Python injects `patch_mode`, `patch_mode_instruction`, and `co_design_draft` as strict required variables; the mode instruction preserves safe/autonomous differences without template conditionals.
- Prompt rendering failures raise `CoDesignAgentError`; there is no hardcoded fallback.
- Codex unavailable and invalid interviewer JSON remain fail-fast.
- `_build_codex_answer_prompt` remains a Python builder and is not migrated.

Phase 4: Redesign Route Migration

- Status: implemented.
- Focused architect/planner/brainstormer route prompts now render `prompts/run/redesign.md` through `metaloop.prompt_pack.render_prompt`.
- Python injects `route_role`, `reviewer_route`, MissionSpec, MissionCapsule, VerificationResult, and SoftReviewDecision as fenced JSON.
- Prompt rendering failures raise `GoalRuntimePromptError`; there is no hardcoded fallback.
- Contract-level redesign semantics, no-edit discipline, locked contract bans, human-acceptance boundary, and required guidance fields remain covered by prompt semantics tests.

Phase 5: Soft Reviewer Migration

- Status: implemented.
- `build_soft_review_prompt` now renders `prompts/run/soft_reviewer.md` through `metaloop.prompt_pack.render_prompt`.
- Python injects MissionSpec, GoalContract, VerificationResult, and the actual `SoftReviewDecision.model_json_schema(by_alias=True)` as fenced JSON.
- Prompt rendering failures raise `SoftReviewPromptError`; there is no hardcoded fallback.
- `CodexSoftReviewer.review` catches prompt render errors and returns a failed low-confidence `SoftReviewDecision` through the existing invalid-output/failure path shape.
- Hard-validator authority, route enum, concrete-worker-fix vs contract/design/decomposition/path-uncertainty routing, actual SoftReviewDecision schema injection, and the human-acceptance boundary remain covered by prompt semantics tests.

Phase 6: Repair Migration

- Next: evaluate repair migration last among the current secondary prompts.
- Preserve locked contract bans, ExecutionReport update requirement, repair attempt state, repeated-repair root cause/hypothesis discipline, and redesign escalation language.

Deferred: Main Goal Prompt

- The main goal prompt is the highest-blast-radius runtime contract.
- It compiles MissionCapsule and GoalContract into the worker's actual authority and acceptance surface.
- It should migrate only after lower-risk prompt families prove the compiler path is stable.

## OpenSpec And Superpowers Borrowed Discipline

From OpenSpec:

- Agree before build: design and acceptance must be explicit before execution.
- Versioned specs: prompt instructions should be inspectable artifacts, not hidden strings.
- Reviewable deltas: future prompt changes should be diffable and testable.

From Superpowers-style workflow discipline:

- Use systematic debugging and repair loops instead of blind retries.
- Require root cause and hypothesis when a repair repeats.
- Keep implementation repair separate from contract redesign.

MetaLoop adapts these ideas to its own core object model. The important artifact is not merely a natural-language prompt; it is the compiled context packet around MissionSpec, Capsule, contract, verification, route, and attempt state.

## Risks And Guardrails

Risk: silent prompt drift.

Guardrail: semantic baseline tests assert durable prompt invariants before migration.

Risk: template language creep.

Guardrail: Phase 1 supports only strict `{{var_name}}` replacement. No expressions, includes, loops, or conditionals.

Risk: missing state masked by fallback prose.

Guardrail: missing files, missing metadata, missing required variables, empty values, and unresolved placeholders are hard errors.

Risk: moving runtime too quickly.

Guardrail: loader is read-only in Phase 1. Runtime builders remain authoritative until each prompt family is migrated with targeted tests.

Risk: weakening locked contracts.

Guardrail: repair and redesign semantics remain separately tested. Repair cannot mutate locked MissionSpec, MissionCapsule, GoalContract, scope, authority, or acceptance. Redesign produces proposal guidance only.

## Why Main Goal And Repair Are Deferred

The main goal prompt is the actual execution contract. A migration bug there could change worker authority, acceptance criteria, or evidence requirements. It should wait until the compiler has passed through lower-risk design-time prompts.

Repair is also deferred because it operates after failure, when state is already fragile. It must preserve locked contract bans, attempt-index discipline, failed-fix summaries, root cause/hypothesis requirements, and ExecutionReport updates. That is too much blast radius for the first compiler integration.

Soft review is deferred because it controls internal routing. A migration bug could turn hard-validator failures into acceptance, ask for human acceptance as an internal route, or choose the wrong repair/redesign path.
